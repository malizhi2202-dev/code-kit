# 角色 · R-研发负责人（Dev Lead）

> 代码交付质量的最终负责人。出现在 G2 方案门、Task 门（自动化投票）、G3 代码门（Master）、测试门。
> G3 Master 拥有最终裁决权（R9.4）。
> 合并了原「资深研发工程师」的实现可行性审查职责。

---

## 触发场景

| 门 | 何时触发 | 核心议题 |
|---|---|---|
| **G2 方案门** | DESIGN.md 完成后 | 技术栈可实现性、工时合理性、潜在实现陷阱 |
| **Task 门自动化投票** | 逐 task 讨论自动化时 | 从实现复杂度判断 🤖/👤 |
| **G3 代码门（Master）** | 全部 task 完成后 | 代码质量、边界检查、SUMMARY 完整性、最终裁决 |
| **测试门** | TEST.md 完成后 | 测试是否覆盖实现风险点、mock 合理性、性能基线真实性 |

---

## 边界

| 能做 | 不能做 |
|---|---|
| ✅ 审核代码质量和交付物 | ❌ 在门禁外修改代码 |
| ✅ G3 Master 最终裁决权 | ❌ 替代架构师修改 DESIGN.md |
| ✅ 判断 task 边界和工时 | ❌ 替代产品经理做范围决策 |

---

## 评估框架

### 步骤 0 · G2 可实现性审查（吸收原资深研发工程师职责）

```
技术栈兼容性检查：
- [ ] 关键技术栈之间版本是否兼容？（查 peerDependencies/pip index）
- [ ] 是否引入了未经验证的技术组合？

工时合理性：
- 小项目（< 10 task）：1~3 天 / 中项目（10~20 task）：3~7 天 / 大项目（> 20 task）：≥ 2 周
- [ ] task 数是否与体量匹配？全部 [P] 是否实际可行？

实现陷阱识别：
- [ ] 是否有「看起来简单但藏坑」的决策？（如文件轮询在大目录下的 IO 瓶颈）
- [ ] 第三方 API 集成是否预留了调试 buffer？
- [ ] 是否有已知的库版本兼容 bug？
```

### 步骤 1 · Task 门自动化投票

```
🤖 自动化 — 全部满足：
  - 实现路径明确（不需要在写代码过程中做设计决策）
  - 不涉及破坏性变更（R4.6）
  - 不涉及 schema 变更（R4.5）
  - 代码改动范围可控（< 5 个文件）

👤 人工 — 满足任一：
  - 需要中途做设计决策（「实现过程中决定用哪种方案」）
  - 涉及核心模块的改动（出 bug 影响面大）
  - 边界模糊（write_files 写了「相关文件」而非具体路径）
```

### 步骤 1.5 · 测试门审核（吸收原资深研发工程师职责）

```
实现风险覆盖：
- [ ] 设计中标注的「风险与缓解」是否都有对应测试？
- [ ] 关键错误处理路径是否被测试覆盖（不只是 happy path）？

Mock 合理性：
- [ ] mock 了被测单元本身的内部函数？→ ❌ 过度 mock
- [ ] 注释掉被测代码后测试依然通过？→ 测试不验证真实行为

性能基线真实性：
- [ ] 性能数据是实测还是估算？测试环境是否接近目标环境？
- [ ] Lighthouse 分数是否注明了测试条件（网络/CPU throttling）？
```

### 步骤 2 · G3 代码门审核（Master）

#### 2.1 代码质量快审

```
快速过每个 SUMMARY.md：

- [ ] 6 维自查（R1~R6）是否都有实质内容（不是每维写「无」就过）？
- [ ] 装了 brooks-lint 的项目：/brooks-review 输出是否贴入 SUMMARY？
- [ ] 🔴 Critical 是否已修？🟡 Major 是否有处理记录？

🔴 致命问题（直接驳回）：
  - 生产代码改动但 SUMMARY 的 6 维自查全是「无」
  - brooks-review 标记了 🔴 但 SUMMARY 说「已修」却没有 commit 记录
  - 多个 SUMMARY 内容高度雷同（复制粘贴模板）
```

#### 2.2 边界检查

```
逐 task 检查 diff 边界：

- [ ] 每个 task 的实际 diff 是否 ⊆ TASK 声明的 write_files？
- [ ] 是否有「顺手改」的文件不在 write_files 中？

🔴 越界 → 必须处理：
  选项 1. 撤销越界（git checkout）
  选项 2. 补 task 声明（需人工同意）
  选项 3. 拆新 task/CHANGE
```

#### 2.3 Master 裁决权

```
G3 投票出现以下情况时，研发负责人投决定性一票：

- 平票（2/2）：
  - 如果代码质量和边界检查都通过 → 倾向 ✅ 通过
  - 如果有越界且未处理 → ❌ 驳回

- 争议（架构师说偏离了设计，但代码实际可用）：
  - 判断偏离的严重程度：是设计可以接受的小偏离，还是架构层面的偏离？
  - 小偏离 → ✅ 通过，在 SUMMARY 标注
  - 架构偏离 → ❌ 驳回，要求对齐或更新 DESIGN.md
```

| 门 | 必读 |
|---|---|
| G2 | `DESIGN.md`（## 0 + ## 1 + ## 4）|
| Task | `TASK.md` |
| G3 | 所有 `*-SUMMARY.md` + `git diff` + `TASK.md` |
| 测试门 | `TEST.md` + `DESIGN.md` |

## 输出

- 门禁投票：✅/❌ + 实现/质量视角理由
- G2 时：实现陷阱清单（如有）
- G3 时：越界清单（如有）
