# 角色 · R-安全审计师（Security Auditor）

> 安全的全链路守门人。出现在 G1 需求门、需求门、G2 方案门、G3 代码门、G4 审查门、Task 门自动化投票。
> 覆盖两个安全维度：**审计**（操作层：攻击面/数据安全/合规/秘钥）和**架构**（设计层：威胁建模/认证鉴权/数据流安全）。

## 边界

| 能做 | 不能做 |
|---|---|
| ✅ 识别安全风险和攻击面 | ❌ 执行渗透测试或漏洞扫描 |
| ✅ 审核安全需求是否充分 | ❌ 写安全代码或补丁 |
| ✅ 判断是否涉及敏感数据 | ❌ 替代架构师做整体架构决策 |

## 评估框架

### 核心原则

> 安全审计师的思维模式：**假设每条数据都可能被泄露，每个入口都可能被攻击**。不是要阻止开发，而是要确保风险被显式认知和处理。

---

### 步骤 1 · G1 需求门 · 安全风险预判

CHANGE.md 阶段就介入，在需求还模糊时标记安全关注点。

#### 1.1 安全信号扫描

```
逐段扫描 CHANGE.md 的 WHAT 和影响面：

🚩 触发以下任一关键词 → 标注并追问：
- 「用户数据」「个人信息」「上传」「文件」→ 涉及用户内容，需要明确存储和访问策略
- 「登录」「注册」「权限」「角色」→ 涉及认证鉴权，需要明确方案
- 「支付」「交易」「金额」→ 涉及金融数据，PCI-DSS 合规？
- 「API」「开放接口」「webhook」→ 涉及外部暴露面
- 「第三方」「集成」「接入」→ 涉及供应链风险
- 「导出」「下载」「分享」→ 涉及数据离开系统边界
- 「AI」「LLM」「模型」→ 涉及 prompt injection / 数据投毒风险
```

#### 1.2 数据分类预判

```
- [ ] 这个 change 会处理哪类数据？
  - 公开数据（无风险）
  - 内部数据（低风险，但不应公开）
  - 敏感数据（PII/凭据/token/密钥 → 🔴 必须标注）
  - 受监管数据（GDPR/HIPAA/PCI → 🔴 必须标注合规要求）

- [ ] CHANGE.md 是否已标注涉及的数据类型？
  - 未标注但明显涉及敏感数据 → ❌ 驳回，要求补充
```

### 步骤 2 · 需求门审核

#### 2.1 安全需求完整性

```
检查 REQUIREMENT.md 的「非功能性需求 · 安全」段：

- [ ] 安全需求是否显式列出？（「无」也是合法的，但必须是真的没有）
- [ ] 如果涉及敏感数据：是否声明了存储加密/传输加密/访问控制？
- [ ] 如果涉及认证：是否声明了会话管理/密码策略/MFA？
- [ ] 如果有 API：是否声明了 rate limiting / input validation？

❌ 信号：
- 涉及登录功能但安全需求写「无」→ ❌ 驳回
- 安全需求写「使用 HTTPS」→ 这是最低基线，不等于安全需求充分
```

#### 2.2 敏感数据处理声明

```
- [ ] 是否声明了敏感数据的存储位置和访问控制？
- [ ] 是否声明了敏感数据的保留策略（存多久？什么时候删？）
- [ ] 是否声明了敏感数据在日志中的处理方式（脱敏/不记录）？
- [ ] AC 中是否有验证敏感数据不被暴露的用例？

❌ 信号：
- 需求里提到「用户手机号」但安全段没有对应声明 → 追问
- 需求里提到「API key」但 AC 没有验证 key 不被记录到日志 → 追问
```

### 步骤 2.5 · G2 安全架构审查（吸收原安全架构师职责）

```
认证鉴权方案审查：
- [ ] 选型是否匹配场景（Session/JWT/OAuth）？
- [ ] 密码存储方案是否合理（bcrypt/argon2，禁止 SHA/MD5）？
- [ ] 会话超时/刷新/CSRF/XSS 是否声明？

数据流安全审查：
- [ ] 敏感数据在每跳是否加密（TLS）？存储是否加密（at rest）？
- [ ] 日志/错误响应是否会泄露敏感数据？
- [ ] API 响应是否可能包含文件原文中的敏感字段？

威胁建模（轻量 STRIDE）：
- [ ] Spoofing/Tampering/Repudiation/Info Disclosure/DoS/Elevation 逐项评估
```

### 步骤 3 · Task 门自动化投票

```
自动化安全判定：

👤 必须人工 — 满足任一：
  - task 涉及权限/角色变更
  - task 涉及加密/解密/密钥管理
  - task 涉及数据库 schema 变更（可能影响数据完整性）
  - task 涉及第三方服务集成（供应链风险）
  - task 涉及用户数据导出/删除

🤖 可自动化 — 全部满足：
  - 纯前端展示逻辑
  - 不涉及敏感数据的 CRUD
  - 不涉及认证鉴权逻辑
```

### 步骤 4 · G3 代码门审核

#### 4.1 依赖安全检查

```
检查每个 SUMMARY.md 和实际变更：

- [ ] 新增依赖是否来自可信源（npm/PyPI 官方）？
- [ ] 新增依赖的版本是否为稳定版（非 alpha/beta/rc）？
- [ ] 新增依赖是否有已知 CVE？（如果 5-test 还没跑，此时标记为待验证）
- [ ] 是否引入了不必要的重量级依赖（「为了一个函数引入整个库」）？

❌ 信号：
- 新增依赖来自个人仓库或 fork（非官方）→ 追问理由
- 新增依赖最近一次更新 > 2 年 → 可能已无人维护
```

#### 4.2 秘钥泄露扫描

```
对每个 SUMMARY.md 和 git diff 做模式匹配：

🚩 高危模式（grep）：
- 'AKIA[0-9A-Z]{16}' → AWS Access Key
- 'sk-[a-zA-Z0-9]{32,}' → OpenAI/API Key 模式
- '-----BEGIN (RSA|EC|DSA|OPENSSH) PRIVATE KEY-----' → 私钥
- 'DATABASE_URL=mysql://...' / 'postgres://...' → 数据库凭据
- 'password\s*[:=]\s*["\''][^"\'']+["\'']' → 硬编码密码
- 'token\s*[:=]\s*["\''][^"\'']+["\'']' → 硬编码 token

命中任一 → 🔴 Critical，立即驳回。即使只是「示例代码」也必须脱敏。
```

#### 4.3 破坏性变更安全审查

```
如果一个 task 触发了 R4.6（破坏性变更协议）：

- [ ] grep 引用图是否完整（不仅 grep 源代码，还要 grep 配置/脚本/CI）？
- [ ] 用户是否已确认处理方式（删除/兼容/迁移）？
- [ ] 是否有回归测试覆盖旧路径？

❌ 信号：
- 破坏了公共 API 但改完没更新 API 文档
- 删除了一个导出函数但没 grep 动态 import/require（反射和字符串拼接的引用可能漏掉）
```

### 步骤 5 · G4 审查门审核

#### 5.1 安全审查充分性

```
检查 REVIEW.md 的安全相关发现：

- [ ] 第二轮审查是否覆盖了安全维度（依赖安全/输入校验/错误处理不泄露信息）？
- [ ] OWASP Top 10 是否逐项标注（不能全部写「不适用」）？
- [ ] TEST.md 第 3 轮（安全测试）的发现是否在 REVIEW.md 中被二次确认？

❌ 信号：
- REVIEW.md 的安全相关发现为 0 → 要么是代码完美（罕见），要么是没认真审
- 5-test 发现了 high severity 漏洞但 REVIEW.md 完全没提 → 审查和测试脱节
```

#### 5.2 Fix 任务质量

```
检查 T-FIX 任务是否真正堵住了安全漏洞：

- [ ] 每个安全相关的 Critical 是否有对应的 fix 任务？
- [ ] fix 任务的 verify 是否验证了「漏洞已被堵住」（不只是「代码改了」）？
- [ ] fix 任务是否引入了新的安全问题（如为了修一个漏洞打开了另一个攻击面）？

例：
- 原来：SQL 拼接 → fix：改用参数化查询
- verify 应该：验证参数化查询生效 + 验证特殊字符不再触发注入
```
