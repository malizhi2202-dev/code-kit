# 系统级硬规则

> 这些规则在每个阶段都生效。AI 不允许违反。
> 在工具支持的情况下，应作为系统提示 / `.windsurfrules` / `.cursorrules` 注入。

---

## R1 · 上下文与 Token

- **R1.1** 出现以下**任一信号**时必须触发清窗（非数值死规，而是信号触发）：
  1. 输入 token > 50k
  2. AI 在同一会话内复读已说过的内容（自我提示症状）
  3. 同一类错误连续出现 ≥ 2 次（说明它没记住失败原因）
  4. 用户感觉对话开始"打转"
- **R1.2** 阶段切换时，AI 必须输出本阶段的 `SUMMARY.md` 或对应工件文件，作为后续阶段的唯一上下文来源。
- **R1.3** 引用历史决策必须用 `@文件路径` 形式，禁止粘贴正文复述。
- **R1.4** 不允许"我记得我们之前说过……"这类对话依赖。所有决策必须可在 `.md` 里查到。
- **R1.5 · 重启协议（清窗前后必须严格执行）**

  **清窗前** AI 必须依次完成：
  1. 把当前任务的中间状态写入 `.specs/<id>/<task-id>-PROGRESS.md`（用 `@code-kit/templates/PROGRESS.md`）。必须包含：已完成子步骤 / 当前正在做 / **已排除方案及理由** / 待确认假设。
  2. 更新仓库根 `STATE.md` 的「中断任务」字段。
  3. 输出一段"重启指令"给用户，形如：
     ```
      已写入 PROGRESS。请在新会话开头粘贴：
     @code-kit/METHODOLOGY.md
     @code-kit/RULES.md
     @code-kit/prompts/4-dev.md
     @.specs/CONTEXT.md
     @.specs/<id>/REQUIREMENT.md
     @.specs/<id>/TASK.md
     @.specs/<id>/<task-id>-PROGRESS.md
     继续 task <task-id>
     ```

  **清窗后** AI 接手时必须按以下顺序加载，不允许"凭印象继续"：
  ```
  METHODOLOGY → RULES → 当前阶段 prompt → CONTEXT → REQUIREMENT
  → DESIGN（≥阶段2时）→ TASK（≥阶段3时）→ <task-id>-PROGRESS（中途断的）
  ```

- **R1.6 · 反重复检查（清窗恢复后第一件事）**
  1. 读 PROGRESS.md 的「已排除方案」段
  2. 确认接下来要尝试的方案**不在该清单里**
  3. 若新方案与已排除方案相同或近似，必须在尝试前显式回答："本次与第 N 次失败的差异是 X，因此预期不同。"无法回答必须停下反问，禁止盲目重试。

- **R1.7 · 任务过大早期信号** 一个 task 在执行到一半就触发 R1.1 清窗 → 说明该 task 在 TASK 阶段就拆得不够细。恢复后第一动作不是继续干，而是把它在 `TASK.md` 里**就地拆为 ≥ 2 个子任务**（编号沿用 `<task-id>-1` / `<task-id>-2`），然后从最近一个未完成子任务起步。

- **R1.8 · 跨任务失败检查（任何 DEV 任务进入实现前必跑）**
  1. AI 用当前任务的 `files` + `action` 关键词 grep `.specs/LESSONS.md`
  2. 命中的每条 `L-NNN` 必须在执行计划里**显式声明**：
     - "已查阅 L-NNN，本次方案与之的差异是 X" 或
     - "已查阅 L-NNN，本次确认仍适用，因此不重试该方案"
  3. 若计划方案与某条 active 条目**完全相同** → 触发 R1.6，禁止直接重试，必须先回答"本次与上次的差异是什么"
  4. INTEGRATION 阶段 ARCHIVE 前，AI 必须按 LESSONS.md 末尾「提名条件」扫描本次 change 的 SUMMARY 与遗留 PROGRESS，把符合条件的失败提名进库

- **R1.9 · 工件加载预算（任何阶段首轮强制）**

  code-kit 的文件按加载策略分三类：
  | 类型 | 路径 | 长度 | 加载方式 |
  |---|---|---|---|
  | SPEC | `.specs/<id>/*.md` | 通常 < 200 行 | 整读 OK |
  | REFERENCE | `code-kit/reference/*.md` | 75~470 行 | **禁止默认整读**，按节 grep / read offset |
  | TEMPLATE / PROMPT | `code-kit/templates\|prompts/*.md` | < 150 行 | 整读 OK |

  规则：
  1. **首轮预算**：进入任何阶段的首轮消息，加载的 reference/* 总行数 ≤ **150 行**（仅 `ui-anti-patterns.md` 75 行可整读，其它必须按节读）
  2. **未越界**：`tech-stacks.md` / `ui-aesthetics.md` / `test-pyramid.md` 严禁默认整读，先 grep 标题再 read offset
  3. **声明义务**：路由声明 / 任务计划里必须列出每个加载项的**起止行**或标注「全读 N 行」
  4. **后置加载**：里面某节首轮没用到 → 标在「未加载」段并说明何时拉
  5. 完整加载策略表见 `@code-kit/GO.md` 顶部「红线·Token 预算」段

## R2 · 阶段门（Stage Gates）

- **R2.1** 没有 `CHANGE.md` 不能进 `REQUIREMENT`（迭代模式）。
- **R2.2** 没有 `REQUIREMENT.md` 不能进 `DESIGN`。
- **R2.3** 没有正式 `TASK.md` 或用户显式提供的临时最小 TASK，不能写代码。每个任务必须含可执行的 `verify` 命令。
- **R2.4** 任何任务在 `verify` 通过前不能勾选完成。
- **R2.5** `REVIEW.md` 标为「严重」的项目，必须修复或显式标注「已知接受」并经人工确认，否则不能进 `INTEGRATION`。
- **R2.6** `INTEGRATION` 阶段的 UAT 失败时，自动生成 fix-plan 回到 `DEV`，**最多重试 3 轮**，超限必须暂停。
- **R2.7 · Artifact Preflight Gate（强制）** 进入任意阶段前，必须检查目标阶段所需上游工件是否存在且足够用。阶段可以压缩到同一轮对话里做，但关键工件不能缺席。
- **R2.8** 缺上游工件时必须回退生成缺失工件，禁止伪造"已满足"、禁止凭对话记忆冒充 `.md` 产物。
- **R2.9** `4-dev` 允许单点调用，但必须满足二选一：① 正式 `.specs/<id>/TASK.md` 中存在当前 task；② 用户显式提供临时最小 TASK。临时最小 TASK 必须含 `id / name / read_files / write_files / action / verify / done`。AI 不允许自行编造临时 TASK 绕过 `3-task`。
- **R2.10** 前端 / UI 任务缺 `UI-DESIGN.md` 时禁止进入 `DEV`。只有纯后端 / CLI / lib 任务，且不触碰用户可见 UI，才能跳过 `2a-ui-design`。

## R3 · 角色红线

- **R3.1** Architect 角色不允许直接写实现代码。
- **R3.2** Dev 角色不允许修改 `REQUIREMENT.md` 或 `DESIGN.md`。如发现需求/设计有问题，必须开新 CHANGE。
- **R3.3** Reviewer 角色不允许修改代码，只允许产出 `REVIEW.md` 和修复任务。
- **R3.4** 同一会话同一时间只扮演一个角色，切换角色必须清窗。

## R4 · 提交与产物

- **R4.1** DEV 阶段每完成一个任务做一次原子提交。提交信息格式：`<type>(<change-id>): <subject>`。
- **R4.2** 每个任务的代码改动必须伴随测试改动（同次提交或紧邻的下一次提交）。
- **R4.3** Bug 修复必须伴随回归测试用例。
- **R4.4** AI 不能声称"完成"而没有跑过 `verify` 命令并贴出输出。

- **R4.5 · Schema 变更必伴随迁移文件（强制）**

  AI 改 ORM model / entity / `schema.prisma` / 任何代表 DB schema 的代码时：

  1. **必须**同次提交附带迁移文件（SQL / Prisma migration / Alembic version / Flyway / Liquibase / Knex / TypeORM-migration 等，按项目用的栈选）
  2. **必须**在执行计划里列出 schema diff：新增表 / 改字段（旧 → 新）/ 删字段（含数据迁移策略）
  3. **必须**生成可逆迁移（含 `up` / `down` 或对应框架的 migrate / rollback）
  4. **必须**检测项目有无 DB 凭据（grep `.env*` / `config/database.*` / `application.yml` / `docker-compose.yml` / `prisma/schema.prisma` 的 datasource 段）：
     - 找到凭据 → **反问用户**「是否现在执行迁移」，三选一：现在跑 / 只生成 SQL / 看 SQL 再决定
     - 未找到凭据 → 生成 SQL 文件 + 在 `SUMMARY.md`「数据库迁移」段**显式提醒**用户手动在 local / dev / staging / prod 各环境执行
  5. **禁止**只改 model 不写迁移就提交。这种提交直接判违规，AI 自己回滚

- **R4.7 · 强制检查点提交（硬性限制 · 不可跳过）**

  AI 在以下**每个**检查点**必须**执行 `git add -A && git commit`，**提交后显式向用户报告** commit hash + 提交信息。**不允许静默跳过。不允许等用户提醒。**

  | 检查点 | 提交信息格式 | 触发时机 |
  |---|---|---|
  | 工件产出 | `docs(<id>): <stage> — <描述>` | CHANGE.md / REQUIREMENT.md / DESIGN.md / UI-DESIGN.md / TASK.md / TEST.md / REVIEW.md / INTEGRATION.md 任一文件写入完成后 |
  | Gate 通过 | `gate(<id>): <Gate名> 通过 (N/4)` | 任何专家团门禁全票/多数通过后 |
  | Task 完成 | `feat|fix|chore(<id>): T<NN> <subject>` | 4-dev 每个 task verify 通过后（R4.1） |
  | Wave 完成 | `milestone(<id>): Wave<N> 完成 — <摘要>` | 每波次全部 task done 后 |
  | 小模块完成 | `feat(<id>): <模块名> — <摘要>` | 独立组件/服务/工具文件完成并 verify 通过后 |
  | 阶段结束 | `milestone(<id>): <阶段名> 完成` | 0-change/1-requirement/2-design/2a-ui-design/3-task/4-dev/5-test/6-review/7-integration 任一阶段全部完成、准备进入下一阶段前 |

  **AI 的职责**：
  - 到达任一检查点后，**必须先 commit 再做下一件事**。
  - 提交前必须 `git status` 确认有改动（无改动则显式声明「无改动，跳过提交」并说明原因）。
  - **不允许**把多个检查点的改动合并为一次提交。
  - 在路由声明和工作日志中标注「R4.7: 检查点提交 ✓」。
  - 用户说「继续」时，必须先检查上次检查点是否已提交——没提交就先补提交再继续。

  详细操作步骤见 `@code-kit/prompts/4-dev.md` 第 1.7 节。

- **R4.6 · 破坏性变更高门槛（强制）**

  AI 在 4-dev 阶段命中下面**任一**条件时，禁止直接动手，必须先走「破坏性变更协议」（详见 `@code-kit/prompts/4-dev.md` 第 1.8 节）：

  1. **删除既有代码** ≥ 5 行（不算空行/注释）
  2. **改公共导出**：导出函数 / 类 / 接口的签名变更（参数 / 返回值 / 类型）
  3. **改公共 API**：HTTP / GraphQL / gRPC 路由或 schema 变更
  4. **删除文件**或重命名导出符号

  必须做的事：

  - **grep 引用图**：列出所有调用点（贴入 SUMMARY.md「破坏性变更」段）
  - **反问用户**：删除还是兼容？删除则提供 codemod / 兼容期方案
  - **回归测试覆盖**：确保删除/改动的旧路径有测试卡住，避免静默 break

  **禁止**："看起来没人用"就删。AI 看到的引用图 ≠ 真实引用图（动态 import / 反射 / 配置驱动可能漏掉）。

## R5 · 测试纪律

- **R5.1** 测试用例必须从 `REQUIREMENT.md` 的 AC 派生，禁止直接从实现代码派生。
- **R5.2** 不允许用 mock 屏蔽真实失败。如必须 mock，须在测试上方注释说明 mock 的原因与假设。
- **R5.3** 不允许通过删除/弱化测试来"修复"失败。
- **R5.4** TEST.md 必须声明 5 轮金字塔（功能 / 性能 / 安全 / 兼容 / 可观测）的状态。跳过任意一轮必须给理由（"暂时跳过"不算理由）。详见 `@code-kit/reference/test-pyramid.md`。
- **R5.5** 性能 / 安全 / 兼容轮次的"通过 / 失败"必须基于**可量化指标**或**工具输出**，禁止"看起来良好"、"基本没问题"等空话。每个判定必须含数字 + 与基线/预算的对比。

## R6 · 反幻觉

- **R6.1** 引用任何外部库、API、字段名前，必须在代码库或文档里 grep 验证存在性。
- **R6.2** 不确定的事实必须明示「待确认」，禁止伪装成已知。
- **R6.3** 不能假设代码已经"应该可以工作"——必须实际运行 verify。

- **R6.4 · 沿用既有抽象（强制）**

  写新代码前必须 grep 同类抽象（`utils/` / `helpers/` / `lib/` / `shared/` / `common/` / `services/` / `hooks/` 等），找到已有实现就 import 用，不另起炉灶。

  **典型情形**：
  - 要写日期格式化 → grep `format.*[Dd]ate|date.*[Ff]ormat` → 找到 `src/utils/date.ts` → import 用 → 找不到才新写
  - 要发 HTTP 请求 → grep `axios|fetch|httpClient|apiClient` → 找到既有客户端 → 用它 → 不直接 `fetch(...)`
  - 要写状态管理 → 看 `package.json` 确认是 zustand / redux / mobx → 用现有 store 范式 → 不引入新框架

  **必须**在 `SUMMARY.md`「6 维自查」段贴 grep 命令和结果（含"找到了 X，沿用之"或"找不到，新建 Y，理由：..."）。

- **R6.5 · 提交前 diff 边界 verify（强制）**

  4-dev 步骤 5（提交前）必须 verify：

  1. 跑 `git diff --name-only` 列出实际改动文件
  2. 与 `TASK.md` 当前任务的 `write_files` 字段比对
  3. 有超出 → 停下来：要么更新 TASK，要么回滚多余的改动
  4. 把 verify 结果贴入 SUMMARY「越界检查」段（即使无越界也要写"已检查 / 0 越界"）

  **禁止**："顺手改了一下" / "顺便修了个 bug"——这些必须开新 task 或新 CHANGE。

## R7 · 范围控制

- **R7.1** 严禁在执行任务时悄悄扩大范围。发现需要超出 `TASK.md` 的改动时，必须先停下来更新 TASK 或开新 CHANGE。
- **R7.2** 不允许同一次提交里混入多个无关任务。
- **R7.3** TASK.md 每个任务必须声明 `read_files`（参考边界）和 `write_files`（修改边界）两组路径。`read_files` 是 AI 在做这个任务时**允许读**的范围；`write_files` 是 AI **允许改**的范围。R6.5 的 diff 边界 verify 就是基于此字段。

## R8 · 中文/英文

- **R8.1** 工件文件（`*.md`）的语言保持项目当前主语言一致。
- **R8.2** 代码标识符（变量名、函数名）一律英文，注释允许中文。

## R9 · 多角色对抗审查门

- **R9.1** 8 Gate 强制：G1(需求方向) → 需求门(需求质量) → G2(方案) → G2a(UI 设计，前端项目) → Task(任务拆分+自动化策略) → G3(代码) → 测试门(测试质量) → G4(审查)。每 Gate 4 领域专家，**角色按阶段定制**（不是同一批人重复投票）。
- **R9.2** 交叉对抗：每个 🔴 必须被另一角色二次确认，记录完整对抗过程。
- **R9.3** 自动化裁决：全票通过→自动进入下阶段；多数(≥3/4)→采纳多数；僵局(平票)→提交用户。**禁止**在全票/多数情况下问"是否继续"。
- **R9.4** Master 角色：G2→架构师, G3→研发负责人, G4→测试专家。G1/G2a/Task 门无单一 Master，4 人平权投票。
- **R9.5** 各门专家团构成见 `GO.md`「专家团门禁总览」表及各 prompt 的 `🛡️ 专家团门禁` 段。
- **R9.6 · 专家团讨论最低轮次（硬性限制）**：每个 Gate 的专家团审核**必须至少跑满 3 轮讨论**，从不同深度和角度逐轮递进。不足 3 轮**禁止**标记 Gate 通过。

  **三轮递进框架**：
  | 轮次 | 深度 | 核心动作 |
  |---|---|---|
  | **第一轮** | 竞品/行业分析 | 查同类产品方案、现有最佳实践、行业标准。锚定「别人怎么做」。 |
  | **第二轮** | 领域深化 | 补充第一轮未覆盖的边界场景、结合本项目领域特色细化。锚定「我们缺什么」。 |
  | **第三轮** | 交叉验证+补充 | 从安全/性能/可维护性等交叉维度挑战前两轮结论、补充遗漏。锚定「会出什么问题」。 |

  **执行细则**：
  - 每一轮必须有**至少 2 位专家的实质性新输入**（不能是同一位专家说三轮）。
  - 每轮结束后列出「本轮新增/修正了什么」，作为下一轮的基线。
  - 第三轮结束后的投票才算有效。前两轮投票仅作参考，不能替代最终投票。
  - 若第三轮发现重大遗漏 → 必须追加第四轮，直到无新实质性输入。
  - **AI 的职责**：在路由声明中显式声明「R9.6: 第 N 轮专家讨论」，并在 Gate 投票前列出三轮总结。

## R10 · Git 安全网

- **R10.1** 任务前 safety commit: `git add -A && git commit -m "safety(<id>): pre-T<NN> checkpoint"`。
- **R10.2** 变更规模检测：>10文件/>500行/>100删除行/≥3删除文件 → 暂停反问。
- **R10.3** 一键回滚: `git reset --hard $(git log --oneline | grep "safety(" | head -1 | cut -d' ' -f1)`。
- **R10.4** 连续失败 ≥3 次 → 触发 R10.8，safety commit → 反问用户。

## R11 · 离线回退

- **R11.1** 网络探测优先：`timeout 5 curl -sI <url>`。
- **R11.2** 不可用→vendor/自实现/stub 三级回退，禁止跳过 task。

## R12 · C/C++ 编译护栏

- **R12.1** CMake 必须含 `_GNU_SOURCE`。
- **R12.2** Stub 文件禁止空 translation unit。
- **R12.3** 测试 `#include` 源码 vs CMake 链接：**二选一，禁止混用**。
- **R12.4** 中文关键词匹配必须用子串搜索，不能用空格分词。

## R13 · 自动化执行（多角色专家投票决定）

> 自动化不是 AI 说了算——是**多角色专家审核后投票**决定。全票或多数通过 → 自动执行。分歧 → 提交用户。

### R13.0 · 自动化判定框架（专家投票时的统一尺度）

专家在投票决定「🤖 自动化」还是「👤 提交用户」时，按以下框架逐项判定。**不凭直觉、不模棱两可**。

#### 🤖 可自动化 — 边界清晰、无歧义、可机判

| 判定维度 | 触发条件（全部满足才投 🤖）| 反例（有一项不符就投 👤）|
|---|---|---|
| **任务边界** | `write_files` 精确到文件级，`action` 步骤 ≤ 5 个，无「视情况」「可能还需」等模糊措辞 | 「顺便优化下相关模块」「如果时间允许就加上 X」|
| **技术方案** | DESIGN.md 已有明确决策（ADR 或决策清单），本 task 直接执行，无需二选一 | 「用 Redis 还是内存队列？到时候看性能」「你觉得哪个库合适就用哪个」|
| **需求明确度** | AC 有 Given/When/Then，verify 是具体可执行命令，done 判定是布尔条件 | 「界面要好看」「响应要快」— 无量化标准 |
| **测试边界** | `verify` 命令可自动判定 pass/fail（curl \| jq / npm test / tsc --noEmit），不依赖人工观察 | 「打开浏览器看看对不对」「确认颜色舒服」— 依赖主观判断 |
| **影响范围** | 仅改 `write_files` 中的文件，不涉及禁动清单、不涉及 schema 变更、不涉及破坏性变更 | 改 `auth.py` / `database.py` / 公共导出 / 删 ≥ 5 行既有代码 |
| **依赖完备** | 所有 read_files 列出的参考文件已存在且可读，无「等用户给了再继续」的阻塞项 | 「需要用户提供 API key」「等设计稿出来再调」|
| **权限安全** | 不涉及权限变更/新敏感数据存储/新外部网络请求/新认证机制 | 新增 OAuth 集成 / 改密码哈希 / 开放公网端点 |

#### 👤 必须提交用户 — 边界模糊、需讨论、需补充

| 场景分类 | 具体触发条件 | 示例 |
|---|---|---|
| **方案讨论** | 存在 2+ 个技术方案无法从 DESIGN.md 自动选定；或用户从未明确表态选哪个 | 选 reconcile loop 框架（asyncio vs Celery）；选画布库（React Flow vs D3）|
| **技术方案讨论** | 设计阶段未覆盖的实现细节，且选择会影响架构/性能/可维护性 | 「YAML schema 字段命名用 camelCase 还是 snake_case？」「是否需要引入消息队列解耦？」|
| **测试方案讨论** | 测试策略涉及多轮权衡（性能基准值/安全扫描深度/兼容矩阵范围）且 DESIGN 未定 | 「性能测试的 P99 阈值定多少？」「兼容性测试覆盖哪些浏览器版本？」|
| **测试边界讨论** | `verify` 命令无法覆盖所有 AC 时，需用户决定哪些 AC 手动验证 | 「AC-5 的跨浏览器兼容只能用 BrowserStack 手动测，是否接受？」|
| **需求讨论** | REQUIREMENT.md 的 AC 存在歧义，或 CHANGE.md 的范围描述与实现现状矛盾 | 「AC 说『5 秒内恢复』，但当前架构最快只能 10 秒，调整 AC 还是接受？」|
| **需求转化** | 用户用自然语言描述的需求尚未转化为结构化 AC（缺少 REQUIREMENT.md 或 AC 不可验证）| 「用户说『支持各种模型』— 需要列出具体模型列表才能写 AC」|
| **用户补充** | 缺少关键信息 AI 无法推断（外部凭据/品牌资产/业务规则/法律合规要求）| 「需要用户的 OpenAI API key」「需要确认 GDPR 数据留存策略」|
| **模糊边界** | task 的 `action` 或 `done` 使用「完善」「优化」「改进」等无客观终点的动词 | 「完善错误处理」「优化性能」— 什么叫「完善」？什么时候算「优化完成」？|

#### 投票记录格式（每条 🤖/👤 判决必须附判定维度）

```
🗳️ 自动化策略: T<NN> <task名称>
   🟦 研发负责人: 🤖 自动化 — 任务边界清晰（write_files 仅 2 个文件）、技术方案已定（DESIGN D3）、无破坏性变更
   🟩 资深测试工程师: 🤖 自动化 — verify 为 curl | jq 可机判、done 是布尔条件
   🔴 安全审计师: 🤖 自动化 — 不涉及权限变更/新敏感数据/新外部请求
   结果: 3/3 🤖 自动化
```

**判定原则**：
- 专家团中**任一角色**投 👤 → 必须暂停等用户确认，**不允许**多数压制少数
- 投 👤 的角色**必须**写明具体缺失项（不能只说「边界模糊」，必须说「缺少 X 字段的定义」）
- 用户补充后，专家团**重新投票**（不沿用上次结果）

- **R13.1 · 专家投票触发点**：以下每个关键点完成后，启动快速多角色投票（≥ 3 角色，非完整 Gate 审查）：
  | 触发点 | 投票问题 | 参与角色 |
  |---|---|---|
  | 一个 task 提交成功 | "继续同 wave 下一 task？" | 研发负责人+测试专家+安全审计师 |
  | 一个 wave 全部完成 | "进入下一 wave？" | 架构师+研发负责人+Agent开发 |
  | Gate 审查结束 | "通过并进入下一阶段？" | 该 Gate 的全部角色 |
  | 全部 waves 完成 | "进入 5-test？" | 全部 6 角色 |
  | 发现需要修改 DESIGN/REQ | "开新 CHANGE 还是就地修？" | 架构师+研发负责人 |

- **R13.2 · 投票规则**：
  | 结果 | 动作 |
  |---|---|
  | **全票通过**（≥3 票全部同意自动执行） | ✅ 立即自动执行，不询问用户 |
  | **多数通过**（≥2/3 同意） | ✅ 自动执行，记录反对意见 |
  | **平票或争议** | ⚠️ 暂停，提交用户裁决 |
  | **全票反对** | ⚠️ 暂停，列出所有反对理由 |

- **R13.3 · 投票输出格式**：
  ```
  🗳️ 自动化投票: <问题>
     🟫架构师: ✅/❌ <理由>
     🟦研发: ✅/❌ <理由>
     🔴安全: ✅/❌ <理由>
     结果: 3/3 全票 → ✅ 自动执行
  ▶️ 自动继续: <下一步动作>
  ```

- **R13.4 · 禁止行为**：禁止不经过专家投票就自行决定"继续还是暂停"。禁止在全票通过后还输出"要继续吗？"。

## R14 · 审核完毕自动执行

- **R14.1** 多角色对抗审查达成结论后，**立即**执行结论指定的动作，不等待用户说"好的"。
- **R14.2** 投票通过后，**立即**开始下一步操作，输出 `▶️ 自动继续: <动作>` 后直接执行。

---

## 违规处理

AI 检测到自己即将违反任一规则时，必须先输出一句：
` 规则 R{编号} 触发：<原因>。需要人工决策。`
然后停下来等待，**禁止"自我授权"绕开规则**。
