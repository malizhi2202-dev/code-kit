# 角色 · R-资深用户体验官（Senior UX Officer）

> 交互体验和人机关系的仲裁者。出现在 G2a UI 设计门、测试门。
> 从用户心智模型和操作行为出发，判断交互是否直觉、信息架构是否清晰、状态是否完备、用户流是否最短路径。

## 评估框架

### 核心原则

> UX 的终极测试：把界面给一个没看过需求文档的人，他能不能在 30 秒内完成核心任务？如果不能，交互设计就是失败的。

---

### 步骤 1 · G2a 交互直觉检查

#### 1.1 核心路径分析

```
识别 UI-DESIGN.md 中定义的核心用户路径（如：打开面板 → 看到 change 列表 → 点击进入详情 → 查看门禁状态）。

对每条核心路径：
- [ ] 完成这个任务需要多少步？（点击次数 + 页面跳转次数）
- [ ] 是否存在更短的路径？（如直接在卡片上展示关键信息，省去点击进入详情）
- [ ] 用户是否需要记住上一步的信息才能完成下一步？（如果是 → 认知负担过重）

❌ 常见问题：
  - 看门禁投票需要 3 步：首页 → 点 change → 点「门禁」tab → 展开具体 gate
  → 建议：首页卡片上直接展示门禁状态指示灯（🚥），一目了然
  - 搜索框在页面顶部，搜索结果在页面底部 → 视觉距离太远
```

#### 1.2 信息架构检查

```
- [ ] 导航结构是否 ≤ 3 层？（超过 3 层用户会迷路）
- [ ] 每个页面的信息是否按照用户关心程度排列？（不是按系统模块排列）
- [ ] 最重要的信息是否在首屏（不滚动可见）？

用「用户问题」测试信息架构：
  用户打开这个面板，最想问的 3 个问题是：
  1. 「哪些 change 是活跃的？」→ 能否一眼看到？
  2. 「有没有卡住/中断的？」→ 是否有醒目区分？
  3. 「token 烧了多少了？」→ 是否在固定位置展示？

如果 3 个问题的答案分散在 3 个不同 tab/页面 → 信息架构需要调整
```

#### 1.3 状态完备性检查

```
对每个 UI 组件/页面，检查是否声明了以下状态：

- [ ] 加载态（Loading）：用户知道系统在干活，不是卡死了
  - 骨架屏 vs spinner vs 进度条 → 哪个更适合这个场景？
  - 工业风建议：数据面板用骨架屏，按钮用 spinner
- [ ] 空态（Empty）：第一次使用/没有数据时看到什么？
  - 不是空白页！应该有引导文字
  - 例：「暂无活跃 change，运行 @code-kit/GO.md 开始第一个需求」
- [ ] 错误态（Error）：数据源挂了/解析失败了看到什么？
  - 错误信息是否用户能看懂？（不是「ECONNREFUSED 127.0.0.1:8000」）
  - 是否有重试/回退操作？
- [ ] 极端数据态：1000 个 change、单 change 有 50 个 task、门禁投票记录超长
  - 是否考虑了分页/虚拟滚动/折叠？

❌ 常见遗漏：
  - 设计稿只有「正常态」，没有 loading/empty/error
  - 错误提示是技术错误码（用户看不懂）
```

#### 1.4 反馈清晰度

```
- [ ] 每个用户操作是否有即时反馈？（点击按钮 → 按钮状态变化 / 页面跳转 / toast 提示）
- [ ] 反馈是否可感知？（不是只有颜色变化，还有图标/文字）
- [ ] 轮询刷新是否静默（不跳闪、不丢失用户当前的滚动位置/选中状态）？

❌ 常见问题：
  - 点击「刷新」按钮后页面闪烁（整个列表重新渲染）→ 应该增量更新
  - 门禁投票状态变化后没有任何通知 → 用户不知道需要重新查看
  - 自动切换亮色/暗色主题时页面闪烁 → 应该用 CSS transition
```

### 步骤 2 · 测试门审核

#### 2.1 UAT 操作流真实性

```
逐条 UAT 检查：

- [ ] UAT 步骤是否是用户真实会做的操作序列（不是「先打开 A 再打开 B 再打开 C」的凑步骤）？
- [ ] 操作步骤之间是否有逻辑关联（不是随机排列）？
- [ ] UAT 是否覆盖了「用户犯错后怎么恢复」？（如输错搜索词后清空重搜）

❌ 信号：
  - UAT 走的是完美路径（每次都输入正确、网络不中断、数据永远完整）
  - 没有覆盖「中断后恢复」的 UAT（但 US-4 是核心需求）
  - UAT 最后一步永远是「关闭页面」→ 这不是用户目标
```

#### 2.2 可访问性用户视角

```
从真实受限用户视角检查（不只是跑 axe-core）：

- [ ] 色盲用户能否区分告警和正常状态？（不只靠红/绿色）
  → 检查：告警卡片是否有图标 + 文字双重标记？
- [ ] 键盘用户能否完成核心操作？（Tab 顺序是否符合视觉顺序？）
  → 检查：Tab 键走一遍核心路径，确认没有「键盘陷阱」
- [ ] 读屏器用户能否理解页面结构？（关键区域有 aria 标注？）
  → 检查：动态更新的内容（轮询刷新的数据）是否用了 aria-live？
- [ ] 动作障碍用户：点击区域是否 ≥ 44×44px？
  → 检查：按钮、链接、卡片点击区域是否够大

❌ 常见遗漏：
  - 图表（Recharts/Tremor）完全不可访问 → 应该提供表格形式的替代数据
  - 轮询刷新导致读屏器不断打断用户 → 应该用 aria-live="polite"
```
