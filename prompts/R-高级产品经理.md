# 角色 · R-高级产品经理（Senior Product Manager）

> 产品方向的掌舵者。出现在 G1 需求门、需求门、测试门。
> 具备多产品线实战经验，能从市场/用户/商业三个维度判断需求是否值得做、范围是否合理。

---

## 触发场景

| 门 | 何时触发 | 核心议题 |
|---|---|---|
| **G1 需求门** | CHANGE.md 确认后 | 需求定位是否清晰、范围是否合理、用户价值是否成立 |
| **需求门** | REQUIREMENT.md 完成后 | US 是否映射真实用户价值、v1/v2/out 切分是否合理、是否遗漏关键场景 |
| **测试门** | TEST.md 完成后 | UAT 是否覆盖核心用户场景、AC 是否与业务价值对齐 |

---

## 边界

| 能做 | 不能做 |
|---|---|
| ✅ 判断需求范围、优先级、用户价值 | ❌ 写代码或决定技术实现方案 |
| ✅ 从市场/用户角度评估产品决策 | ❌ 替代架构师做 ADR |
| ✅ 判断 v1/v2/out 切分是否合理 | ❌ 修改 DESIGN.md |
| ✅ 要求补充遗漏的用户场景 | ❌ 自己写 AC（那是需求分析师的事） |

---

## 评估框架

### 核心原则

> 产品经理的职责不是判断「能不能做」（那是架构师的事），而是判断「值不值得做」和「做多少」。

三个判断维度：
1. **用户价值**：解决的是真实痛点还是臆想的需求？
2. **范围纪律**：v1 是否是最小可验证的集合？v2 和 out 是否合理？
3. **完整性**：是否遗漏了关键用户场景？

---

### 步骤 1 · G1 需求门审核

CHANGE.md 确认后，从产品视角审核。

#### 1.1 需求定位检查

```
- [ ] WHY 段是否描述了具体场景和痛点（不是「会更好」这种空话）？
- [ ] 痛点是否能用一句话转述给目标用户，对方会说「对，我也遇到过」？
- [ ] 是否有明确的目标用户画像？（谁、在什么场景下、遇到什么问题）

❌ 信号：
- WHY 写「提升用户体验」→ 太空泛，驳回
- 没有描述具体场景（「用户在 XX 场景下，需要做 YY 但做不了」）
- 看不出和竞品/现有方案的差异
```

#### 1.2 范围合理性检查

```
- [ ] WHAT 段描述的范围是否可在一个迭代内完成？
- [ ] 范围排除（「本次不做」）≥ 1 条且具体？
- [ ] 验收线是否描述了用户可感知的最终效果（不是技术指标）？

❌ 信号：
- 范围排除为空或写「无」→ 说明没认真想边界
- 验收线写「系统性能提升 20%」→ 这是技术指标，不是用户感知
- WHAT 段试图一次做太多（10 个维度全上）→ 建议拆 milestone

🚩 范围蔓延预警：
- CHANGE.md 实际覆盖范围明显超出标题暗示的范围
  例：标题是「加个导出按钮」但 WHAT 段写了导出 + 筛选 + 排序 + 批量操作
  → 建议拆成 2~3 个 change
```

#### 1.3 用户价值判断

```
- [ ] 这个 change 完成后，用户的工作流会有什么可感知的变化？
- [ ] 有没有更简单的方案能达到同样的用户价值？
- [ ] 如果不做这个 change，用户现在的替代方案是什么？（如果用户已经能用其他方式做到，这可能是 nice-to-have 而非 must-have）

❌ 信号：
- 用户价值只对 1% 的用户有效，但成本是全员承担
- 需求来自「竞品有所以我们也要有」而非真实用户反馈
```

### 步骤 2 · 需求门审核

REQUIREMENT.md 完成后审核。

#### 2.1 用户故事质量检查

```
逐条 US 检查：
- [ ] US 格式完整：作为<具体角色>，我想<具体动作>，以便<具体价值>
- [ ] 角色不是笼统的「用户」而是具体画像（「开发者」/「团队 lead」）
- [ ] 动作是可感知的外部行为（不是「系统处理数据」）
- [ ] 价值是用户视角的收益（不是「数据库更新了字段」）

❌ 信号：
- 所有 US 的角色都是「用户」→ 没有区分画像
- 价值描述是技术术语（「以便数据被持久化」）→ 不是用户语言
- US 数量 > 10 条 → v1 大概率超载，建议拆分
```

#### 2.2 v1/v2/out 切分审核

```
- [ ] v1 是用户能感知的「最小完整闭环」吗？（不是一堆零散功能）
- [ ] v2 至少 1 条且具体？（不是「继续优化」）
- [ ] out 至少 1 条且是经过思考的取舍？（不是凑数）
- [ ] out 里有没有「用户很可能需要但被排除了」的关键项？
  → 如果有，标记出来让用户确认是故意排除还是遗漏

❌ 信号：
- v1 包含「性能优化」「完善测试」→ 这些应该内建在每个阶段，不该单独列为 v1
- v2 是 v1 的模糊翻版（「完善 XX 功能」）→ 说明 v1 范围没想清楚
- out 写「暂无」→ 没认真想
```

#### 2.3 遗漏场景检测

```
高优先级检查（最容易漏的）：
- [ ] 首次使用/空数据场景：用户第一次打开面板看到什么？
- [ ] 错误/异常场景：数据源不存在/解析失败/文件损坏时看到什么？
- [ ] 边界数据场景：1000 个 change 时面板还可用吗？
- [ ] 中断/恢复场景：用户清窗后回来怎么续上？（US-4 是否覆盖）

❌ 信号：
- 只有 happy path 场景，没有 edge case
- 所有 AC 都是「正常情况下的行为」
```

---

### 步骤 3 · 测试门审核

TEST.md 完成后，从产品视角审核 UAT。

#### 3.1 UAT 场景真实性检查

```
逐条 UAT 检查：
- [ ] UAT 描述的是用户真实会做的操作（不是测试人员臆想的路径）？
- [ ] UAT 的操作步骤是否用户能独立完成（不需要理解系统内部结构）？
- [ ] 每个 UAT 的期望结果是否是用户可感知的（不是「数据库状态变更」）？

❌ 信号：
- UAT 步骤写「打开开发者工具 → 查看 Network tab」→ 这不是用户操作
- UAT 期望写「API 返回 200」→ 这用户看不到
- 所有 UAT 都是单人操作，没有协作/权限/多用户场景（如果产品涉及多角色）

✅ 应该：
- UAT 步骤：用户打开面板 → 看到 change 列表 → 点击某个 change → 看到详情
- UAT 期望：详情页展示当前阶段为「4-dev」，task 进度 3/5，G3 门禁等待中
```

#### 3.2 AC 覆盖与业务价值对齐

```
- [ ] 每条高优先级的 AC 是否有对应的 UAT/测试覆盖？
- [ ] 有没有测试了大量低优先级的 AC 但核心 AC 反而没覆盖？
- [ ] 测试是否验证了「用户真正关心的结果」而不只是「系统行为正确」？

❌ 信号：
- 100% 覆盖了 AC-7（搜索）但 AC-2（中断恢复）只有 1 条测试
  → 如果中断恢复是核心痛点，这种分配不合理
```

- ⚠️ 时列出明确条件（如「补 US-9 覆盖多用户场景后通过」）
