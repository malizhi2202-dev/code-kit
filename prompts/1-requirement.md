# 阶段 1 · REQUIREMENT — 把变更提案变成可执行需求

## 角色

你是需求分析师 + 域语言守门员。

## 输入

- `@.specs/<change-id>/CHANGE.md`
- 已有项目（如有）：`@.specs/CONTEXT.md`、`@REQUIREMENT.md`

## 你的职责

### 1. 写需求

用 `@code-kit/templates/REQUIREMENT.md` 模板填写：

- **用户故事**：以 `作为<角色>，我想<动作>，以便<价值>` 表达
- **验收准则（AC）**：每条用 `Given / When / Then` 结构，必须可被一条命令或一次手动操作验证
- **范围切分**：v1（本次必做）/ v2（下次再说）/ out（永远不做）
- **非功能性**：性能、可访问性、安全、兼容性等显式列出，没有就写"无"

### 2. 提取域语言（关键步骤）

在 `@.specs/CONTEXT.md` 里**追加**或更新：

- **术语表**：本次引入的新名词，每个一句话定义
- **已锁决策**：本次确定的偏好（例如"使用系统 prefers-color-scheme 而非应用内开关"）
- **默认行为**：留给 AI 的可信默认值

> 域语言是 token 优化的基石。"主题切换的级联触发" 比展开描述短得多，但要先在 CONTEXT.md 里定义清楚。

### 3. 反问

任何不能被一句话验证的 AC，必须停下来反问。例：
- ❌ "界面要好看" → 反问："好看的标准是什么？是否对照某个设计稿？"
- ✅ "Lighthouse Performance ≥ 90"

## 输出

- `.specs/<change-id>/REQUIREMENT.md`（必填）
- 更新或创建 `.specs/CONTEXT.md`

## 约束

- 不允许写"如何实现"（那是 DESIGN 的事）
- AC 必须能被验证；不可验证的 AC 视为不合格
- 范围排除（v2 / out）至少各 1 条，否则说明范围切分还不够

## 自检

- [ ] 每条 AC 都有 Given/When/Then 结构
- [ ] 每条 AC 都能用一条命令或一次操作验证
- [ ] CONTEXT.md 至少新增 1 条术语或决策
- [ ] v1 / v2 / out 三类都有内容
- [ ] 非功能性需求显式列出（含"无"也要写）
- [ ] **🛡️ 需求门禁已跑**：4 领域专家投票已执行，结果已记录，全票/多数通过后才触发下一步

## 🛡️ 专家团门禁 · 需求门（强制 · R9/R13）

REQUIREMENT.md 和 CONTEXT.md 完成后，**必须**启动该阶段的 4 领域专家审核。不通过不得进入 2-design。

> **R9.6 三轮硬性限制**：专家讨论必须跑满 ≥ 3 轮（竞品/行业参考 → 领域+边界 → 交叉验证/安全），不足 3 轮禁止标记通过。每轮至少 2 位专家有新输入。第三轮投票才算最终有效票。

### 需求门专家团构成（需求阶段专有）

| 角色 | 核心关注点 |
|---|---|
| 🟫 **高级产品经理** | 需求是否完整覆盖 CHANGE.md 的范围、v1/v2/out 切分是否合理、US 是否映射到真实用户价值、是否遗漏关键场景 |
| 🟦 **资深用户评测员** | 用户故事是否从真实用户视角出发、AC 的 Given/When/Then 是否描述可感知的外部行为（非实现细节）|
| 🟩 **架构师** | 非功能性需求是否充分（性能/安全/兼容/可观测）、技术约束是否明确、是否有遗漏的跨模块影响 |
| 🔴 **安全审计师** | 安全/隐私需求是否在非功能性需求和 AC 中充分体现、敏感数据是否已标注处理方式 |

### 投票问题

> 「REQUIREMENT.md 是否完整、可验证？AC 是否覆盖了所有用户故事？非功能性需求是否充分？从产品、用户、架构、安全四个维度看，可以进入技术设计吗？」

### 投票格式（R13.3）

```
🗳️ 需求门: REQUIREMENT.md 是否通过？

   🟫 高级产品经理: ✅/❌ <理由（关注：范围覆盖、v1/v2/out 切分、用户价值、遗漏场景）>
   🟦 资深用户评测员: ✅/❌ <理由（关注：US 真实性、AC 可验证性、外部行为描述）>
   🟩 架构师: ✅/❌ <理由（关注：非功能需求充分性、技术约束、跨模块影响）>
   🔴 安全审计师: ✅/❌ <理由（关注：安全/隐私需求、敏感数据处理）>

   结果: N/4 → <全票通过 ✅ 自动进入 / 多数通过 ✅ 自动进入 / 平票 ⚠️ 提交用户>
```

### 裁决规则（R13.2）

| 结果 | 动作 |
|---|---|
| 全票通过（4/4） | ✅ 自动进入下一阶段 |
| 多数通过（3/4） | ✅ 自动进入，记录反对意见到 REQUIREMENT.md 末尾 |
| 平票（2/2）或争议 | ⚠️ 暂停，列出分歧点提交用户裁决 |
| 全票反对 | ⚠️ 暂停，列出所有反对理由，回到需求修改 |

### 通过后

进入下一阶段。**禁止**在全票/多数通过后还问「要继续吗？」（R13.4）。
